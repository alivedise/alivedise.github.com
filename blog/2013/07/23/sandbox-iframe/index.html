Modules Inside Iframe

by Alive Kuo - Jul 23rd, 2013 -

Recently we had some discussions about modulization. One of those is: Proposition of an architectural change for Gaia apps.

The main idea is load the modules into iframe, containing:

  • HTML file
  • The remaining are loaded as well as the iframe
    • js files
    • css files
    • l10n resources (it is, the dictionary relavant to the frame.)

First thing comes to my mind is, we would end up with a “lots of iframe” HTML, which just looks like frameset…;)

My real concerns:

  1. iframe load latency - We may encounter UI flash before the content is fully loaded.
  2. postMessage overhead - To communicate with other modules we couldn’t use previous window.dispatchEvent nor ‘pub/sub’. We will need to use postMessage very often.
  3. network overhead - We could group all every templating files into one but it seems we couldn’t merge the HTML files in difference into big one.

The good parts I am heard:

  1. This way enforce people to separate the logic. (But people could use a big iframe as well.)
  2. Some JS bytecodes are shared between frames. We don’t need to parse it again.
  3. We could unload an iframe entirely if we don’t need it anymore.

The last one looks to me the most critical one.

Any MVC pattern couldn’t guarantee that if you destroy a module, the relative parts(DOM/js) are thrown right away.

IMO I really wish one day we have the ability to make sure GC is done not only on iframes but on the DOM elements we want it to.

AI hope that we won’t end up that only one framework, one library is left for choice.
